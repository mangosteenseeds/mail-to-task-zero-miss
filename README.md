# メール起点・業務漏れゼロ タスク管理システム（テンプレート）

> 受信メールからタスク化・検品差戻し管理・業務漏れゼロまでを
> **n8n + Google Sheets + Slack + Claude API** で自動化する、汎用オープンソーステンプレート。
>
> このリポジトリは**設計パターンとアーキテクチャのみ**を提供する。特定の導入先・顧客・個人の情報は一切含まない。実運用時は、自社・顧客固有の情報（顧客名簿、担当者、SLA実値など）を別途プライベートに管理すること。

## これは何か / 誰のためか

継続的な受託業務（Web制作・保守、制作代行、士業の顧問業務、運用代行など）で、**メールで依頼が飛んでくる小〜中規模チーム**向けの、タスク管理自動化テンプレート。

次のような状態に効く：

- 受信ボックスで依頼が埋もれ、タスク化されないまま漏れる
- 要件すり合わせのメール往復で、タスクのゴールが曖昧なまま停滞する
- 「今日、何件・何時間分の作業があるか」が担当者に見えない
- 業務終了時に「終わってない案件」「返信忘れ」「検品待ち放置」が漏れる
- 自社対応待ちか、依頼者の応答待ちかが見えず、停滞が見過ごされる
- 納品後の修正依頼（差戻し）が繰り返され、気づいたら期日を超過する
- マネージャーの判断待ちで業務全体が止まる（意思決定の単一障害点化）

「動く最小構成」ではなく **「業務漏れゼロが担保できる最小構成」** を MVP と定義しているのが特徴。

## 設計の中核思想（このテンプレの肝＝転用価値の高い概念）

ここがこのリポジトリで一番価値のある部分。技術スタックは置き換え可能だが、以下の概念は業種を問わず転用できる。

### 1. スレッドベース状態遷移モデル
「メール1通 = タスク1件」ではなく **「メールスレッド = 1案件」** として扱う。要件すり合わせの往復を1案件の状態遷移として管理し、情報の分散を防ぐ。

### 2. 段階的タスク化（readiness ベース）
受信＝即タスク化、ではない。AIが**要件成熟度（readiness score）**を評価し、閾値を超えた時のみタスク化する。要件が固まる前のスレッドは「要件確認中」として別管理し、その停滞も検知する。

### 3. 検品 = 別ステータス / 差戻し = 子タスク
- 納品時点では `delivered`（検品待ち）。依頼者の承認受領で初めて `closed`。
- 差戻し（修正依頼）は親タスクの status を変えず、**独立した子タスク**として記録する。
- `closed` 後の追加メールは「蒸し返し」ではなく**新規案件**として別スレッド扱い。

### 4. ★ ball_holder（ボール所在）— 停滞検知の核
すべてのタスクは、いま誰にボールがあるかを必ず持つ。

| ball_holder | 意味 | 例 |
|---|---|---|
| `us` | 自社が対応すべき状態 | open / in_progress / 返信ドラフト承認待ち |
| `customer` | 依頼者の応答待ち | delivered（検品待ち）/ 要件確認返信待ち |
| `system` | システム・外部待ち | AI処理中 / 申請中 / 外部承認待ち |

状態遷移のたびに `ball_since` を更新。これにより「自社が24h以上ボールを持ったまま」「依頼者が3日応答していない」が**機械的に**検知できる。`ball_holder` / `ball_since` は単一の真実の情報源であり、あらゆる遷移で必ず更新する（漏れると停滞検知が動かない）。

### 5. ★ Push型UI ＋ 業務終了の明示化（漏れゼロの仕組み）
- **Pull型ではなく Push型**：担当者が見にこなくても通知が能動的に届く。
- **Daily Briefing**：始業時、当日の業務量・優先順位・期日リスクを担当者DMに配信。
- **EOD Checklist**：終業前、未完了・未承認・未確認をすべて列挙。担当者が「業務終了」ボタンを押すまでリマインドが繰り返される。
- 業務終了は**時刻ではなくボタン押下**で確定する（人間の意思で締める）。

### 6. ★ 意思決定キュー（マネージャー・ボトルネック対策）
「マネージャー判断待ち」を業務全体の最終ボトルネックと捉え、構造化する。

- 経営判断を要する案件はすべて `decision_queue` に登録される。
- urgency 別の **SLA タイムライン**に基づき、段階的にリマインドが発火する。
- 主決裁者が未応答なら、副決裁者へ自動エスカレ。
- それでも未応答なら、**最終エスカレ通知（高優先度・電話要請を含む）**が発火する。
- 事前ルールで定めた範囲のみ、SLA最終超過時に**自動デフォルト判断**を実行する。
- ただし **顧客との関係性に関わる判断（謝罪・期日交渉など）は絶対に自動化しない**。

## アーキテクチャ

### 技術スタック

| レイヤー | 技術 | 役割 |
|---|---|---|
| メール基盤 | Gmail (Google Workspace) | 依頼受信。特定ラベルでフィルタ |
| ワークフロー基盤 | n8n（セルフホスト or Cloud） | 全自動処理のオーケストレーション |
| AI | Anthropic Claude API | メール分類・要件成熟度評価・返信ドラフト生成・原因分析 |
| データベース | Google スプレッドシート | threads / tasks_current / 各種ログ |
| 運用 UI | Slack | 通知・ボタン操作・Daily Briefing・EOD Checklist |
| 可視化（任意） | Looker Studio 等 | 月次ダッシュボード |

> スプレッドシートをDBに使うのは小規模前提の割り切り。規模が出たら RDB（Postgres など）に差し替えれば良い。設計概念はそのまま転用できる。

### データフロー（受信〜タスク化）

```
Gmail（対象ラベル）
   │  1分ポーリング
   ▼
n8n Gmail Trigger
   ▼
WF-00 ノイズフィルタ（自動通知・除外送信元を弾く）
   ▼
WF-01 メール処理 → Claude API で分類・成熟度評価
   ▼
threads シート更新 / ball_holder 設定
   ▼
[分岐] readiness 閾値超 → tasks_current にタスク生成 → Slack 通知
        閾値未満         → 要件確認中として保持（停滞も監視）
```

## データモデル（主要シート / 抜粋スキーマ）

実運用の列は導入先に合わせて増やすが、転用可能な最小核は以下。

**threads（案件＝スレッド単位の状態）**
`thread_id` / `subject` / `customer_id` / `assignee` / `status` / `readiness_score` / `ball_holder` / `ball_since` / `last_message_at`

**tasks_current（現在進行中のタスク）**
`id` / `thread_id` / `title` / `status`（open / in_progress / delivered / closed）/ `assignee` / `estimated_hours` / `actual_hours` / `deadline` / `final_deadline` / `ball_holder` / `ball_since` / `parent_id`（差戻し子タスク用）/ `rework_round_count` / `carryover_count`

**decision_queue（判断待ちの一元管理）**
`id` / `task_id` / `category` / `urgency`（critical / high / normal）/ `requested_at` / `reminder_count` / `escalation_level` / `status` / `decided_by` / `decided_at`

**ログ系**
`audit_log`（重要操作の記録）/ `daily_briefing_log` / `eod_checklist_log` / `system_health_log`

> 顧客名簿・担当者マッピング・除外送信元・各種SLA実値などは、このリポジトリには含めず **`config/*.csv` のプレースホルダ**として用意し、実値はプライベートに管理する。

## Slack UI パターン

- **Daily Briefing（DM, 始業時）**：当日タスク（工数付き）・優先順位・期日リスク。`業務開始` ボタンで `work_started_at` 記録。
- **タスク通知（チャンネル）**：`着手する` / `納品完了` 等のボタンで status 遷移。至急案件は専用チャンネル＋影響シミュレーション（期日リスク・本日残工数）。
- **EOD Checklist（DM, 終業前）**：未完了・検品未確認・承認待ち・未着手を列挙。実工数入力（0.5h刻み）。持ち越し理由は必須選択。`業務終了確定` を押すまでリマインド継続。
- **意思決定エスカレ**：判断待ちを決裁者DMに通知 → SLA に沿って段階リマインド → 副決裁者へエスカレ → 最終エスカレ通知。

## 意思決定エスカレ（SLA ラダー / 役割ベース）

役割は固有名ではなく**ロール**で定義する。

| ロール | 説明 |
|---|---|
| `PRIMARY_APPROVER`（主決裁者） | 全カテゴリの判断 |
| `BACKUP_APPROVER`（副決裁者） | 主決裁者の不在・SLA超過時の代行判断、最終エスカレ先 |

**SLA タイムライン（例・環境変数で調整）**

| urgency | 初回 | リマインド1 | リマインド2 | 副決裁者エスカレ | 最終エスカレ通知 |
|---|---|---|---|---|---|
| critical | 即時 | +30分 | +1時間 | +2時間 | +3時間 |
| high | 即時 | +2時間 | +4時間 | +1営業日 | +2営業日 |
| normal | 即時 | +4時間 | +1営業日 | +2営業日 | +3営業日 |

**自動デフォルト判断**は `config/decision-rules.csv` に明示したものだけ実行（コードにハードコードしない）。顧客関係性に影響する判断は対象外。

## セットアップ

1. **`.env` を用意**（下記プレースホルダ参照）
2. **n8n 接続確認**
3. **スプレッドシート初期化**：必要シートを作成し、ヘッダー行と `config/*.csv` の初期データを投入
4. **Phase 1 実装**（業務漏れゼロ基準）：WF-00 / WF-01（簡易版）/ WF-04 / WF-05 / WF-08 / WF-09 / WF-10 / WF-12
5. **n8n へデプロイ**
6. **テスト**：`tests/sample-emails/`（匿名サンプル）＋ Daily Briefing / EOD Checklist の動作確認

**実装の優先順位**：①業務漏れゼロを最優先（機能を簡略化しても漏れ検知は省略しない）→ ②動くことを優先 → ③既存スタックを優先 → ④シンプルを優先 → ⑤判断不能なら人間に確認。

## 実装上の注意

- **スプレッドシート競合対策**：Sheets ノードは Lookup → Update。row index ではなく `id` 列で検索更新する。
- **Claude API レート制限**：Exponential backoff（1s / 2s / 4s、最大3回）。
- **冪等性**：同一 `message_id` の二重処理を防止。EOD 等のリマインドは `reminder_count` で重複防止。
- **個人情報・機微情報の取り扱い**：機微情報を含むメールは本文を要約のみ表示し、原文はメーラのリンク経由で参照。Anthropic の Zero Data Retention オプション利用を検討。
- **監査ログ**：返信送信・タスク化・ステータス変更・業務終了確定など重要操作は `audit_log` に記録。
- **自己監視**：ヘルスチェック自身が落ちる可能性に備え、外部監視（UptimeRobot 等）で n8n の死活確認を併設。

## このテンプレートに含まないもの（意図的に除外）

- 特定の導入先・顧客・取引先の名称、件数、構成比などの業務データ
- 実在の担当者・決裁者の氏名や評価
- 業種固有（医療・士業など）の運用詳細やコンプライアンス判断
- 本番の SLA 実値・自動判断ルールの中身

これらは導入先ごとに**プライベートに**管理する。本リポジトリは「同じ問題を持つ人が、ゼロから作らずに済む」ための骨格のみを提供する。
